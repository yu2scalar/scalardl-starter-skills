# scalardl-starter-skills

A Claude Code plugin that bundles interactive skills for bringing up ScalarDL applications. Each skill is a Q&A-driven slash command that emits a working artefact (config bundle, Spring Boot starter project, Contract / Function JSON definitions, …).

## Skills

| Skill | Role | Status |
|---|---|---|
| `/scalardl-generate-config` | Server-side deploy + runtime client bundle: Helm values × 4 (Ledger / Auditor / 2 schema-loaders) + properties × 3 (ledger.as.client / auditor.as.client / runtime client) + ECDSA P-256 PKI (DS) or HMAC shared secret + License PEM injection + self-signed TLS certs with SAN. Targets ScalarDL 3.13.0+. Authentication patterns: "all DS" or "all HMAC" (mixed configurations are deliberately not supported). | Ready |
| `/scalardl-generate-springboot-starter` | Spring Boot 3.5 / Java 17 / Gradle scaffold with the ScalarDL Java Client SDK wired into a service layer, REST endpoints for Contract / Function register + execute, Swagger UI at `/swagger-ui/index.html`. Auto-detects an adjacent `scalardl-generate-config` bundle (Branch A — consume host / port / auth / cert / HMAC from it) or runs standalone Q&A (Branch B — DS or HMAC chosen explicitly). Supports DS and HMAC authentication as first-class options. Bakes literal sample-grade secrets into `application.properties` + `cert/` with a production-migration warning block + `.gitignore` enforcement. | Ready |
| `/scalardl-add-contract` | Add one Contract JSON definition to a project generated by `scalardl-generate-springboot-starter`. Q&A picks a preset (`READ_ASSET` / `PUT_ASSET` / `PUT_ASSETS`), gathers asset-id fields and a sample input, writes `examples/<Name>.json`. The running app registers it via `POST /api/contracts/register`. | Ready |
| `/scalardl-add-function` | Add one Function JSON definition to a project generated by `scalardl-generate-springboot-starter`. Q&A picks a preset (`UPSERT_RECORD` / `UPSERT_RECORDS`), gathers the linked Contract id, the ScalarDB schema, and column → input mappings, writes `examples/<Name>Function.json`. The running app registers it via `POST /api/functions/register`. | Ready |
| `/scalardl-validate-config` | Audit an existing ScalarDL configuration (Ledger / Auditor / Client properties + Helm values) against a 16-rule catalog. Detects Auditor-vs-proof requirement violations, mixed-config traps, TLS pairing gaps, deprecated keys, cross-file mismatches (HMAC secret / authentication.method), and AI-leverage lints (typo / enum violation / homoglyph identifiers). Read-only — reports findings with severity, recommendation, and reference-doc citations. | Ready |

Each skill is launched on its own; they communicate through an on-disk `.scalardl-starter-skills.json` manifest at the project root, written by `scalardl-generate-springboot-starter` and read by the `add-*` skills.

## Install

```bash
# At the Claude Code prompt
/plugin marketplace add /path/to/scalardl-starter-skills
/plugin install scalardl-starter-skills@scalardl-starter-skills
/reload-plugins                # or restart the session
```

After reload, the slash commands above are available.

```
│  └─ marketplace.json
├─ .gitignore
├─ README.md
└─ plugin/
   ├─ .claude-plugin/
   │  └─ plugin.json
   └─ skills/
      ├─ scalardl-generate-config/
      ├─ scalardl-generate-springboot-starter/
      ├─ scalardl-add-contract/
      ├─ scalardl-add-function/
      └─ scalardl-validate-config/
```

The marketplace `source` points at `./plugin/`, so only the `plugin/` subdir is shipped at install time.

## Design principles

- **Audience**: Java / Spring Boot developers picking up ScalarDL for the first time, plus operators bringing up a Ledger.
- **Don't substitute silently**: when an answer conflicts with a ScalarDL constraint (Contract immutability, the Function `Database` API limit, signed vs unsigned arguments, …), the skill surfaces the trade-off and lets the user decide.
- **Self-contained references**: each skill carries the docs it needs under `references/`. Skills do not read external paths at runtime.
- **Working baseline**: scaffolds aim to be *runnable*, not just typecheck-clean. After scaffolding, ask Claude Code for follow-on edits — the structure is ready for them.
