# scalardl-add-function

> Status: **v0.3.0 (2026-05-01 — Preset matrix consolidated to UPSERT_RECORD / UPSERT_RECORDS. Targets ScalarDL 3.13.0+ exclusively; runtime compiler emits Java 17 bytecode.)**
> Related plans: plan-002 (skill restructure + presets), plan-007 (preset consolidation + rename). Plan docs live in the repo under `docs/` for contributors.

## Overview

Adds **one Function JSON definition** to an already-scaffolded project. The skill emits a single `examples/<Name>Function.json`; the running app turns it into a registered Function via `POST /api/functions/register`.

Functions run **atomically with their linked Contract**: when the client calls `/api/contracts/execute`, both writes commit (or roll back) together. The Function reads the same JSON the Contract received via `contractArgument`, so you don't pass a separate body for the Function — the column → input field mapping below is what determines which fields land in which ScalarDB column.

Run `scalardl-generate-springboot-starter` first; this skill does not create a project.

## Pre-flight check

Same as `scalardl-add-contract`: read `.scalardl-starter-skills.json` at the current working directory.

- Missing → tell the user to run `scalardl-generate-springboot-starter` first. Stop.
- Present → parse `packageName` for the default Java package.

Optionally also enumerate `examples/*.json` looking for Contract definitions (anything with an `op` field). Use the discovered Contract names as candidates for the **linked Contract id** question (C0 below).

## Preset comparison cheat-sheet

| Preset | When to use | Generated body |
|---|---|---|
| **UPSERT_RECORD** | Write or overwrite a single ScalarDB row. Pairs with single-asset Contracts (`PUT_ASSET`). The generated `.java` ships a commented-out `getContractContext` sample showing how to consume a Contract-computed value. | `database.put(put)` returning `{status: "upserted", table}` |
| **UPSERT_RECORDS** | Write-or-overwrite N rows in one Contract execute. Pairs with multi-asset Contracts (`PUT_ASSETS`). Iterates `contractArgument.assets[]`; per-entry validation + per-entry `database.put`. Same homogeneous-shape constraint as the linked Contract. | Iterates `contractArgument.assets[]`; per-entry `database.put(put)`. Returns `{status: "upsertedAll", table, count}`. |

> **About the preset names** (v0.3.0): `_RECORD` (singular) = one ScalarDB row; `_RECORDS` (plural) = N rows in one Contract execute. The verb is `UPSERT` (not `PUT`) because ScalarDB's `Put` class is being deprecated in favour of explicit `Insert` / `Upsert` / `Update`; ScalarDL's Function-side exposure of those is under consideration. The current generated body still calls `database.put(put)` (the only mutator the Function `Database` interface exposes today), but `database.put` enforces an implicit pre-read so the storage-layer behaviour is upsert regardless. Naming the preset `UPSERT_RECORD` matches the future ScalarDB direction so it survives the eventual API migration.

Other ScalarDB ops (UPDATE / DELETE / READ from inside a Function) are not preset — fork the template at `templates/src/main/resources/function-templates/` and ask Claude Code to populate it.

### Pairing recommendations (auto-suggested in C0a below)

| Linked Contract op | Suggested Function preset |
|---|---|
| `READ_ASSET` | (none — read-only Contracts typically don't pair with a Function) |
| `PUT_ASSET` | `UPSERT_RECORD` |
| `PUT_ASSETS` | `UPSERT_RECORDS` |

## Q&A flow

### Step 1 — Pick a preset

```
Pick a Function preset:
  1) UPSERT_RECORD   (write-or-overwrite a single row; pair with PUT_ASSET Contracts)
  2) UPSERT_RECORDS  (write-or-overwrite N rows; pair with PUT_ASSETS Contracts)
```

### Step 2 — Common inputs

| # | Question | Notes |
|---|---|---|
| C0 | Linked Contract id | the registered class name of the Contract this Function pairs with, e.g. `InventoryV1_0_0`. Auto-discover candidates by scanning `examples/*.json`; let the user override. |
| C0a | Suggest preset based on linked Contract's op | After C0, read the linked Contract's `op` from `examples/<linked>.json`. Suggest the matching Function preset per the pairing table above (linked Contract `PUT_ASSETS` → suggest `UPSERT_RECORDS`; `PUT_ASSET` → suggest `UPSERT_RECORD`). The user can override; the skill must not silently substitute a different preset. |
| C1 | Function base name | default: `<linkedContractBase>Function` (e.g. `InventoryFunction`) |
| C2 | Version | default matches the linked Contract |
| C3 | Java package | default: `<packageName from manifest>.functions` |
| C4 | Paste the **ScalarDB schema.json** for the target table | accepts JSON literal or file path. Must contain exactly one `namespace.table` entry with `partition-key`, optional `clustering-key`, and `columns`. |
| C5 | Map each column → input field | for every column in the schema, pick one of the input fields from the linked Contract's `sampleInput` (or, for `PUT_ASSETS` Contracts, from each entry of the `assets` array). Required for every column. |

### Step 3 — Confirm

Echo a JSON definition preview and ask for go-ahead. Once confirmed, write the file at `examples/<Name>Function.json`.

### Step 4 — Offer auto-preview (optional)

After writing the JSON file, ask the user:

> JSON is in. Want me to render `<Name>Function.json` to `generated/functions/<Class>.java` now? Two equivalent paths:
>
>   1. **Online** — `POST /api/functions/preview` (requires `./gradlew bootRun` running at `http://localhost:8080`).
>   2. **Offline** — `./gradlew render -Pargs="<Name>Function"` (no Spring Boot, no Ledger; just the Mustache pipeline).
>
> Pick one, or skip. (default: skip)

If **Online**:

1. Run:
   ```bash
   curl -sS -X POST http://localhost:8080/api/functions/preview \
     -H 'Content-Type: application/json' \
     -d @examples/<Name>Function.json
   ```
2. If the curl succeeds (HTTP 200): summarise — print `className`, `javaPath`, and a short snippet (or full source) of `javaSource`. Draw extra attention to the commented-out `getContractContext` block at the top of the rendered `invoke` — that's the typical first edit point when the linked Contract emits a computed value.
3. If the curl fails (connection refused, validation error, etc.): say so plainly, suggest the offline path (`./gradlew render -Pargs="<Name>Function"`) as a fallback that doesn't need the app, and stop. Do not auto-retry.

If **Offline**:

1. Run:
   ```bash
   ./gradlew render -Pargs="<Name>Function"
   ```
2. If the task succeeds: read `generated/functions/<Class>.java` and display it. Point at the commented-out `getContractContext` block as the typical first edit point when the linked Contract emits a computed value (same callout as the online path).
3. If the task fails (exit code != 0, or stderr mentions `[skipped]`): show the stderr line plainly. Common causes: column name in `mappings` doesn't match `scalarDbSchema.columns`; `partition-key` / `clustering-key` references a missing column; type unsupported.

If **Skip**: go directly to the completion message.

This step is interactive because both paths are no-op against the Ledger but require either a running app (Online) or a built classpath (Offline). The user knows which one their environment supports.

## Output — single JSON file

```json
{
  "name": "<C1>",
  "version": "<C2>",
  "packageName": "<C3>",
  "op": "<preset>",
  "linkedContractId": "<C0>",
  "scalarDbSchema": <C4>,
  "mappings": <C5>
}
```

If `examples/<C1>Function.json` already exists, **do not overwrite**. Tell the user "a Function with that name exists — bump the `version` or pick a different base", then ask for C1 and C2 again.

## Completion message

```
✅ examples/<Name>Function.json written for preset <preset>.

Two paths to register it (with the app running):

A) 1-shot — render + compile + register on the Ledger:
  curl -X POST http://localhost:8080/api/functions/register \
    -H 'Content-Type: application/json' -d @examples/<Name>Function.json

B) Preview → edit → register (especially useful when you want to consume the linked
   Contract's setContext() value):
  curl -X POST http://localhost:8080/api/functions/preview \
    -H 'Content-Type: application/json' -d @examples/<Name>Function.json
  # → edit generated/functions/<Name>FunctionV1_0_0.java, e.g. uncomment + adapt
  #   the getContractContext sample
  curl -X POST http://localhost:8080/api/functions/register-from-source \
    -H 'Content-Type: application/json' -d '{"className":"<Name>FunctionV1_0_0"}'

Apply the ScalarDB schema (one-time, before executing):
  scalardb-schema-loader --config <your-scalardb-config> --schema-file <your-schema.json>

Execute (the Function runs atomically with its linked Contract):
  curl -X POST http://localhost:8080/api/contracts/execute \
    -H 'Content-Type: application/json' \
    -d '{
      "contractId": "<C0>",
      "argument": <a payload matching the Contract's sampleInput>,
      "functionId": "<Name>V<version>",
      "functionArgument": "{}"
    }'
```

Both presets print this heads-up:

```
Heads-up: the generated .java carries a commented-out getContractContext sample
showing how to consume a value the linked Contract computed via setContext(...).
If the linked Contract (PUT_ASSET or PUT_ASSETS) emits a computed value, ask
Claude Code to uncomment + adapt that sample after registering once, then bump
`version` in this JSON and register again.
```

## Limitations

1. **One Function per skill run.** Run again to add another.
2. **One linked Contract.** Multi-Function pairing per Contract execution is a v0.2 candidate (see Plan 002 §11).
3. **`functionArgument` is unused.** Clients always pass `"{}"` for it; all Function inputs come through `contractArgument`.
4. **No mapping-aware validation today.** If a column listed in the schema has no entry in `mappings`, the running app fails at register-time with a clear error — re-run this skill with the missing mapping filled in.

## ⚠ Heads-up — sharing the Function-written table with external ScalarDB apps

The table this Function writes is a regular ScalarDB table, so **another ScalarDB application can read or write it directly**. If you plan that dual-access pattern, **both the ScalarDL Ledger and the external ScalarDB app must point at the same Coordinator table** — otherwise transaction commit state cannot be reconciled across the two apps, and reads may return stale or uncommitted data.

This is a `scalar.db.*` configuration concern, not a Contract / Function code concern. The official warning lives in ScalarDL's "How to run applications" doc:

> If your applications read and write a table through the Function feature, and the table is also directly accessed from ScalarDB applications, you need to properly configure the database chosen here. Specifically, **both ScalarDL and ScalarDB applications must refer to the same Coordinator table to guarantee consistency.**
> — `https://scalardl.scalar-labs.com/docs/latest/how-to-run-applications#underlying-database`

**Keys that must match** between the ScalarDL Ledger's `scalar.db.*` (configured by the `scalardl-generate-config` skill, Phase 5) and the external ScalarDB app's config: `scalar.db.storage`, `scalar.db.contact_points`, and the Coordinator namespace (default `coordinator`). The Auditor's ScalarDB stays independent — this requirement only constrains the Ledger ↔ external app pair.
