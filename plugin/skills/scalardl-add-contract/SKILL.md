# scalardl-add-contract

> Status: **v0.3.0 (2026-05-01 — Preset matrix consolidated to READ_ASSET / PUT_ASSET / PUT_ASSETS. Targets ScalarDL 3.13.0+ exclusively; runtime compiler emits Java 17 bytecode.)**
> Related plans: plan-002 (skill restructure + presets), plan-007 (preset consolidation + rename). Plan docs live in the repo under `docs/` for contributors.

## Overview

Adds **one Contract JSON definition** to an already-scaffolded project. The skill emits a single `examples/<Name>.json` file under the current project; the running Spring Boot app turns that JSON into a registered Contract via `POST /api/contracts/register`.

This skill does **not** scaffold anything from scratch — run `scalardl-generate-springboot-starter` first.

## Pre-flight check

The skill must be run from inside a project directory scaffolded by `scalardl-generate-springboot-starter`. To detect that, read `.scalardl-starter-skills.json` at the current working directory:

```bash
$ ls -la ./.scalardl-starter-skills.json
```

- **If the file is missing**: tell the user "this skill must run inside a project scaffolded by `scalardl-generate-springboot-starter`. Run that skill first, then come back." Stop.
- **If the file is present**: parse it and pre-fill the defaults below from these fields:
  - `packageName` — used as `<projectPackage>` in the defaults table
  - `projectName`, `auditorEnabled`, `scalardlSdkVersion` — useful for the completion message and documentation

## Preset comparison cheat-sheet

| Preset | When to use | Generated `invoke` shape |
|---|---|---|
| **READ_ASSET** | Read one asset by id. Default returns the latest entry only; pass `historyLimit > 1` (or `0` for unlimited) in the runtime argument to get history. Optional `ageOrder` / `startAge` / `endAge` knobs supported. One registered Contract handles every query variant. | `ledger.scan(AssetFilter)` returning `{count, asset_records: [{id, age, data}, ...]}` (array shape regardless of count) |
| **PUT_ASSET** | Write or modify one asset. Absent → writes a fresh asset; present → runs a `// TODO (modify)` block (default fallback overwrites with input) and `ledger.put` the result. ScalarDL Ledger appends a new aged entry on every put — old entries persist (audit chain). | `ledger.get` → branch by `existing.isEmpty()`; both branches end in `ledger.put(assetId, ...)` |
| **PUT_ASSETS** | Atomic write/modify across **two or more** assets that share the same data shape (homogeneous). Use cases: balance transfer between accounts, material → product transformations, bulk updates. | Reads all assets first, then a `// TODO (multi-asset modify)` block, then `ledger.put` each back. Atomic via the in-flight Ledger transaction. |

If none of the three fits, ask Claude Code to extend the generated `.java` after a register call (the file at `generated/contracts/<ClassName>.java` carries `// TODO` markers for that). Note: `.java` is regenerated on every `/register`, so use a fresh `version` if you want to keep the edits.

> **About the preset names** (v0.3.0): `_ASSET` (singular) = one `assetId`; `_ASSETS` (plural) = multiple `assetId`s in one execute. `READ_ASSET` returns a `{count, asset_records:[...]}` array even when count = 1, so callers parse one shape regardless of `historyLimit`. `PUT_ASSET` / `PUT_ASSETS` are named after `ledger.put` (stable Ledger API) — note that `ledger.put` is **append-only versioning**, NOT row-replacement. Old asset entries persist forever for the audit chain.

## Q&A flow

### Step 1 — Pick a preset

```
Pick a Contract preset:
  1) READ_ASSET   (read by id; default latest, optional history)
  2) PUT_ASSET    (write/modify single asset, with TODO modify block)
  3) PUT_ASSETS   (atomic write/modify across N homogeneous-shape assets)
```

### Step 2 — Common inputs

| # | Question | Notes |
|---|---|---|
| B1 | Paste the **input JSON example** (the runtime payload your client will POST to `executeContract`) | accepts JSON literal or file path |
| B3 | Contract base name | e.g. `UserAccount`, `Inventory` |
| B4 | Version | default `1.0.0` |
| B5 | Java package | default: `<packageName from manifest>.contracts` |

For preset **READ_ASSET** / **PUT_ASSET**:

| # | Question | Notes |
|---|---|---|
| B6 | Which input fields make up the asset id? | multi-select; joined with `:` |

For preset **PUT_ASSETS**, the input JSON in B1 must contain an `assets` array of homogeneous-shape entries. Then:

| # | Question | Notes |
|---|---|---|
| B6m | Which fields **inside each `assets[]` entry** make up that entry's asset id? | multi-select; joined with `:`; applied to every entry in the array |

### Step 3 — Confirm

Echo a JSON definition preview matching the chosen preset and ask for go-ahead. Once confirmed, write the file at `examples/<Name>.json`.

### Step 4 — Offer auto-preview (optional)

After writing the JSON file, ask the user:

> JSON is in. Want me to render `<Name>.json` to `generated/contracts/<Class>.java` now? Two equivalent paths:
>
>   1. **Online** — `POST /api/contracts/preview` (requires `./gradlew bootRun` running at `http://localhost:8080`).
>   2. **Offline** — `./gradlew render -Pargs="<Name>"` (no Spring Boot, no Ledger; just the Mustache pipeline).
>
> Pick one, or skip. (default: skip)

If **Online**:

1. Run:
   ```bash
   curl -sS -X POST http://localhost:8080/api/contracts/preview \
     -H 'Content-Type: application/json' \
     -d @examples/<Name>.json
   ```
2. If the curl succeeds (HTTP 200, JSON response with `javaPath` / `javaSource`): summarise — print `className`, `javaPath`, and a short snippet (or full source) of `javaSource`. Mention the file is on disk at `javaPath` so the user can edit it directly.
3. If the curl fails (connection refused → app not running; HTTP 500 → schema validation error; etc.): say so plainly, suggest the offline path (`./gradlew render -Pargs="<Name>"`) as a fallback that doesn't need the app, and stop. Do not auto-retry.

If **Offline**:

1. Run:
   ```bash
   ./gradlew render -Pargs="<Name>"
   ```
2. If the task succeeds: summarise — print the path that the CLI logged (`generated/contracts/<Class>.java`) and read the file's content to display the rendered Java to the user.
3. If the task fails (exit code != 0, or stderr mentions `[skipped]`): show the stderr line plainly, and stop. Same Mustache pipeline as `/preview`, so failures here mean the JSON itself has a problem (e.g., unknown column name in mappings).

If **Skip**: go directly to the completion message.

This step is interactive because both render paths are no-op against the Ledger but require either a running app (Online) or a built classpath (Offline). The user knows which one their environment supports.

## Output — single JSON file

The skill writes one file at `examples/<Name>.json` (where `<Name>` is the answer to B3). Shape varies slightly by preset:

### READ_ASSET / PUT_ASSET

```json
{
  "name": "<B3>",
  "version": "<B4>",
  "packageName": "<B5>",
  "op": "<preset>",
  "assetIdFields": [/* B6 */],
  "sampleInput": <B1>
}
```

For `READ_ASSET`, the `sampleInput` is for asset id derivation + documentation only — runtime scan parameters (`historyLimit`, `ageOrder`, `startAge`, `endAge`) are passed by the client at execute time, not stored in the JSON. Default `historyLimit=1` (latest only); pass `0` for unlimited.

### PUT_ASSETS

```json
{
  "name": "<B3>",
  "version": "<B4>",
  "packageName": "<B5>",
  "op": "PUT_ASSETS",
  "assetIdFields": [/* B6m, applied to each entry of sampleInput.assets */],
  "sampleInput": <B1, must contain "assets": [...]>
}
```

If `examples/<Name>.json` already exists, **do not overwrite**. Tell the user "a definition with that name exists — bump the `version` field or pick a different `<Base>` name", then ask for B3 and B4 again.

## Completion message

```
✅ examples/<Name>.json written for preset <preset>.

Two paths to register it (with the app running):

A) 1-shot — render + compile + register on the Ledger in one call:
  curl -X POST http://localhost:8080/api/contracts/register \
    -H 'Content-Type: application/json' -d @examples/<Name>.json

B) Preview → edit → register (for custom logic without committing to the Ledger first):
  curl -X POST http://localhost:8080/api/contracts/preview \
    -H 'Content-Type: application/json' -d @examples/<Name>.json
  # → look at generated/contracts/<Name>V1_0_0.java, edit by hand or ask Claude Code:
  #     "edit generated/contracts/<Name>V1_0_0.java to ..."
  curl -X POST http://localhost:8080/api/contracts/register-from-source \
    -H 'Content-Type: application/json' -d '{"className":"<Name>V1_0_0"}'

Execute it (after register):
  curl -X POST http://localhost:8080/api/contracts/execute \
    -H 'Content-Type: application/json' \
    -d '{"contractId":"<Name>V1_0_0","argument":<a payload matching sampleInput>}'

Want to pair a Function (atomic ScalarDB write on each Contract execution)?
  /scalardl-add-function
```

For **PUT_ASSET** and **PUT_ASSETS** presets, additionally print:

```
Heads-up: the generated <Name>V<...>.java carries a `// TODO (modify)` block.
Default fallback overwrites the asset with the input. To add real modify logic
(merge, arithmetic, preconditions), ask Claude Code to edit the generated file
after registering once, then bump `version` in this JSON and register again.
The `.java` is regenerated on every /register call, so version-bumping is what
keeps your edits durable.
```

## Limitations

1. **One Contract per skill run.** Run the skill again to add another.
2. **Homogeneous multi-asset only.** Heterogeneous shapes (e.g. `from` and `to` with different fields) are out of scope; fork the template if needed.
3. **Behaviour is JSON-shape-only.** The skill never encodes computational logic (arithmetic, branching, validation expressions) into the JSON — that's delegated to Claude Code editing the generated `.java`. See `feedback_logic_delegation_to_claude.md` for why.
